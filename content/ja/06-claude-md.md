# 08. CLAUDE.md 完全ガイド

> Claude Codeの長期記憶を設定する最も重要なファイル、CLAUDE.mdをマスターしましょう。

---

## CLAUDE.mdとは？

Claude Codeはセッションが終わると会話内容を忘れてしまいます。次に起動したときには、初対面のように何も覚えていません。

**CLAUDE.md**はこの問題を解決します。

Claude Codeは起動するたびにCLAUDE.mdファイルを自動で読み込み、プロジェクトの背景やルールを把握した状態で作業を開始します。

CLAUDE.mdは、Claudeに渡す**指示書であり、プロジェクト説明書**です。

```
あなた: 「このプロジェクトはNext.jsで作っていて、TypeScriptを使う。
       タブではなくスペース2つでインデントし、日本語でコミットメッセージを書いて。」

→ これを毎回伝える代わりに、CLAUDE.mdに一度書いておけば永続的に記憶されます。
```

---

## どこに作る？

CLAUDE.mdファイルの配置場所によって、適用範囲が変わります。

| ファイルの場所 | 適用範囲 | チーム共有 |
|----------|---------|------------|
| `./CLAUDE.md` | 現在のプロジェクト | ✅ Gitでチーム共有可能 |
| `./.claude/CLAUDE.md` | 現在のプロジェクト（隠しフォルダ） | ✅ Gitでチーム共有可能 |
| `./CLAUDE.local.md` | 現在のプロジェクト（自分専用） | ❌ .gitignoreに自動追加 |
| `~/.claude/CLAUDE.md` | すべてのプロジェクト（グローバル） | ❌ 自分のみ適用 |

**最も一般的な場所**: プロジェクトルートの `CLAUDE.md`

```bash
# プロジェクトフォルダでCLAUDE.mdを作成
touch CLAUDE.md

# またはClaude Codeに自動生成してもらう
/init
```

> `/init` コマンドを実行すると、Claude Codeが現在のプロジェクトを分析してCLAUDE.mdの下書きを自動生成してくれます。

## 何を書けばいい？

CLAUDE.mdに決まったフォーマットはありません。Markdownで自由に書けます。以下の項目を含めると特に効果的です。

> 💡 ここでいう「プロジェクト」は開発プロジェクトに限りません。**リサーチフォルダ、コンテンツ制作フォルダ、業務自動化フォルダ**など、あらゆる作業フォルダにそのまま適用できます。Claude Codeはコードがないフォルダでも同じように動作します。

### 必須項目

#### 1. 作業の概要
Claudeがこのフォルダで何をすべきか把握できるよう、簡潔に説明します。

```markdown
## 作業の概要
このフォルダは競合他社の市場調査リサーチプロジェクトです。
調査結果は reports/ フォルダにMarkdownで保存します。
```

#### 2. ツール／環境
使用するツール、資料、保存方式を明記します。

```markdown
## 作業環境
- **主要ツール**: Notion（企画）、Google Docs（下書き）、Canva（デザイン）
- **資料の保存場所**: docs/ フォルダ（Markdown形式）
- **言語**: 日本語優先、英語資料は要約翻訳
```

#### 3. 作業ルール
文章のスタイル、ファイル命名規則、出典の基準などを指定します。

```markdown
## 作業ルール
- 文体: 丁寧かつ明確に（誇張表現は禁止）
- ファイル名: YYYY-MM-DD_テーマ.md 形式
- 出典のない統計・数値は使用禁止
- 常に日本語で記述
```

#### 4. やってはいけないこと（禁止事項）
ミスを防ぐため、禁止事項を明示しておくと効果的です。

```markdown
## 絶対にやらないこと
- 出典のない統計の引用
- 未検証の情報を事実として記述
- 顧客・パートナーの実名をドキュメントに記載
- 旧バージョンのファイルを上書き（常に日付入りファイル名を使用）
```

#### 5. よく使う作業手順
Claudeによく依頼する作業や、繰り返しワークフローを定義します。

```markdown
## よく使う作業
- リサーチレポート作成: 「要約 → 重要インサイト5つ → 出典リスト」の形式で
- 議事録整理: 「決定事項 → アクションアイテム（担当/期限） → 未解決事項」の形式で
- ドキュメントレビュー: まず要約を提示し、修正が必要な箇所は理由とともに記載
```

## 実例: 目的別CLAUDE.mdテンプレート

自分の作業タイプに合ったテンプレートを選び、コピーしてすぐに使えます。

---

### テンプレートA ── リサーチワークスペース

```markdown
# リサーチプロジェクトのルール（CLAUDE.md）

## プロジェクトの目的
- 目標: [調査テーマと目的を1行で]
- 成果物: report.md（要約）、sources.md（出典）、slides-outline.md（プレゼン用目次）

## 資料／フォルダのルール
- 原文資料は sources/ フォルダに保存、ファイル名: YYYY-MM-DD_出典_タイトル.md
- メモ・仮説は notes/ フォルダに保存（原文と混在させない）

## 出典ルール
- すべての主張に出典を付ける（リンク＋根拠を1行）
- 出典の信頼度: A（公的機関/一次情報）/ B（業界/二次情報）/ C（個人ブログ/意見）
- 不確かな場合は「推定・仮説」と明記

## 文章／書式
- 常に日本語、文は短く
- 成果物の基本構成: 1)5行の要約 2)重要インサイト5つ（根拠付き） 3)反論・リスク3つ 4)次のアクション3つ
```

---

### テンプレートB ── コンテンツ制作ワークスペース

```markdown
# コンテンツ制作のルール（CLAUDE.md）

## ブランドトーン
- 親しみやすいが誇張しない（「完璧/革命/絶対」は禁止）
- 読者: [ターゲット読者の説明]、専門用語は括弧で簡潔に解説

## チャネル別フォーマット
- ブログ: タイトル3案 → 本文（小見出し4〜6個） → チェックリスト → CTA 1つ
- SNS: 7〜9枚のカード型投稿（1枚につき1文）、最後のカードにまとめ＋リンク
- ニュースレター: 3行の要約 → 本文600〜900文字 → 次回テーマの予告

## 禁止／注意事項
- 競合他社の実名は記載禁止（必要な場合は「A社/B社」と匿名化）
- 統計・数値は出典がなければ「例示」と表示

## ファイル出力ルール
- 下書き: drafts/YYYY-MM-DD_テーマ_チャネル.md
- 最終版: final/ フォルダに保存、先頭にバージョン/レビュアー/公開日のメタデータを記載
```

### テンプレートC ── ビジネス運営ワークスペース

```markdown
# 運営ドキュメントのルール（CLAUDE.md）

## 会社／プロダクトの背景
- プロダクト: [名称] / 顧客: [ターゲット] / コアバリュー: [1行で]
- よく使う用語の定義: [用語: 定義] の形式で下に蓄積

## コミュニケーションスタイル
- メール/メッセージ: 丁寧語、3段落構成（要約 → 詳細 → 次のステップ）
- クレーム対応: 「共感の1文 → 事実確認の質問2つ → 解決策の選択肢2つ」

## ドキュメントテンプレート
- 議事録: 会議の目的 / 決定事項 / アクションアイテム（担当/期限） / 課題
- 提案書: 問題定義 → 解決策 → スケジュール/範囲 → コスト → リスク

## セキュリティ／機密情報
- 個人情報・決済情報はマスキング（例: 090-****-1234）
- 外部共有前に「機密表記/顧客名」の自動チェックリストを先に出力
```

## グローバル設定: すべてのプロジェクトに適用する

`~/.claude/CLAUDE.md` ファイルを作成すると、すべてのプロジェクトに共通で適用されるルールを設定できます。

```bash
# グローバルCLAUDE.mdを編集
nano ~/.claude/CLAUDE.md
```

グローバルCLAUDE.mdに書くとよい内容：

```markdown
# グローバル設定（全プロジェクト共通）

## 個人の好み
- 説明は常に日本語で
- コード変更前に、何をするか先に説明して
- 重要な変更は実行前にもう一度確認して

## コーディングスタイル（共通）
- 関数はできるだけ短く（20行以内）
- 変数名は意味が明確になるように
- コメントよりも、コード自体を読みやすく
```

## メモリの優先順位

複数のCLAUDE.mdファイルがある場合、より具体的なファイルが優先されます。

```
優先順位（高 → 低）:
1. CLAUDE.local.md（個人のプロジェクト設定）
2. ./CLAUDE.md（プロジェクト設定）
3. ~/.claude/CLAUDE.md（グローバル設定）
4. 組織管理のCLAUDE.md（会社設定）
```

たとえば、グローバル設定で「スペース2つ」にしていても、プロジェクトのCLAUDE.mdで「タブ使用」に設定すれば、プロジェクト設定が優先されます。

## 便利なTips

### `/memory` コマンドの活用
セッション中にCLAUDE.mdを直接編集したい場合：

```shell
/memory
```

ファイル選択画面が開き、CLAUDE.mdをその場で編集できます。

### ファイルの参照（Import）
CLAUDE.mdから他のファイルを参照できます：

```markdown
# プロジェクト指針

プロジェクト概要は @README.md を参照してください。
利用可能なnpmスクリプトは @package.json を参照してください。

## 追加ルール
- Gitワークフロー: @docs/git-guide.md
```

### まずはCLAUDE.mdから始めよう！

新しいプロジェクトを始めるなら、**最初の作業としてCLAUDE.mdを作りましょう**。

```shell
# プロジェクト開始時にClaude CodeにCLAUDE.mdの自動生成を依頼
/init
```

あるいは、Claudeに直接お願いすることもできます：

```
「このフォルダは競合他社の市場調査リサーチプロジェクト。
 成果物は reports/ フォルダにMarkdownで保存して、
 すべての主張に出典を付けて。日本語で書いて。
 この内容をもとにCLAUDE.mdファイルを作って。」
```

> **開発者の場合**: 「このプロジェクトはNext.js 14 + TypeScript + Tailwindで作る。パッケージマネージャーはpnpm。CLAUDE.mdファイルを作って。」

## まとめ: CLAUDE.md早見表

| 項目 | 内容 |
|------|------|
| **役割** | Claudeの長期記憶／プロジェクト指示書 |
| **場所** | プロジェクトルートの `CLAUDE.md` |
| **グローバル設定** | `~/.claude/CLAUDE.md` |
| **自動生成** | `/init` コマンドを使用 |
| **編集** | `/memory` コマンドまたは任意のテキストエディタ |
| **書式** | 自由なMarkdown |

**CLAUDE.md 1つで毎回の説明の手間をなくし、Claude Codeを本当のチームメンバーのように使いこなしましょう。**

---

CLAUDE.mdをしっかり書いておくだけで、毎セッションの説明が不要になります。まるで本当のチームメンバーのように使えるようになります。
