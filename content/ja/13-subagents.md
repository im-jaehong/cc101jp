# 17. Sub-agents & Agent Teams

> Claudeが別のClaudeに作業を委任 — 複雑な作業を並列処理する方法

---

## Sub-agentsとは？

**Sub-agents（サブエージェント）** は、メインのClaudeが特定の作業を専門化された別のClaudeインスタンスに委任する機能です。

> **簡単なたとえ**: チームリーダーがプロジェクトを受け取ったら、一人で全部やるのではなくメンバーに分担します。「リサーチはAに、翻訳はBに、最終校正はCに」 — Sub-agentsはClaudeがこのように作業を分担する仕組みです。

各サブエージェントは：
- **独立したコンテキストウィンドウ**を持ちます（メインの会話と分離）
- **カスタムシステムプロンプト**で特定の役割に集中します
- **制限されたツール**のみ使用します（必要なものだけ）
- 作業完了後、**要約結果のみ**をメインに返します

---

## 概念図

```
メインClaude（オーケストレーター）
    │
    ├──► サブClaude A: 探索担当（Explore）
    │         └── ファイル読み込み、コード検索のみ可能
    │
    ├──► サブClaude B: 実装担当（general-purpose）
    │         └── ファイル読み込み+修正可能
    │
    └──► サブClaude C: テスト担当（custom）
              └── Bash実行のみ可能

各サブエージェントは独立コンテキストで実行
→ メインコンテキストをクリーンに保つ
→ 結果の要約のみメインに返す
```

---

## いつ役立つか？

### Sub-agentsを使うと良い場面

| 場面 | 理由 |
|------|------|
| 大量の出力が生じる作業 | テストログやドキュメント取得結果がメインコンテキストを圧迫する |
| 独立した複数ファイルの同時修正 | 並列処理すればより高速 |
| 特定のツールだけを使う作業 | 読み込み専用レビュアー、DBクエリ専用エージェント |
| 探索と実装を分離したい時 | 探索結果がメインの会話を汚染しないように |

### Sub-agentsより通常の会話が良い場面

| 場面 | 理由 |
|------|------|
| 頻繁なフィードバックが必要な作業 | サブエージェントは独立実行 → 途中介入が困難 |
| 素早い簡単な修正 | サブエージェント起動のオーバーヘッドがある |
| 複数ステップがコンテキストを共有すべき時 | 計画 → 実装 → テストが密接に関連している場合 |

## 組み込みサブエージェント

Claude Codeはデフォルトで3つのサブエージェントを含んでいます：

| エージェント | モデル | 許可ツール | 用途 |
|---------|------|----------|------|
| **Explore** | Haiku（高速） | 読み込み専用 | ファイル探索、コード検索 |
| **Plan** | メイン継承 | 読み込み専用 | Planモードでのコンテキスト収集 |
| **general-purpose** | メイン継承 | すべてのツール | 複雑なマルチステップ作業 |

Claudeがコードを探索する必要がある時は、自動的に**Explore**サブエージェントを使います。探索結果（数千行のファイル内容）がメインの会話を汚染せず、要約だけがメインに返されます。

## 独自のサブエージェントを作る

### 方法1: /agents インタラクティブメニュー（推奨）

```
/agents
```

1. **Create new agent**を選択
2. **User-level**（すべてのプロジェクト）または**Project-level**（現在のプロジェクト）を選択
3. **Generate with Claude**を選択して役割を説明
4. 許可ツールを選択
5. モデルを選択（Haiku/Sonnet/Opus）
6. 保存

### 方法2: ファイルの直接作成

以下の例を`.claude/agents/`フォルダに保存すれば、すぐに使用できます。

**リサーチ要約エージェント**（`.claude/agents/researcher.md`）：

```yaml
---
name: researcher
description: 資料フォルダを読み、要点の要約・インサイト・次の質問を整理します。リサーチが必要な時に自動で使用。
tools: Read, Grep, Glob
model: sonnet
---

あなたはリサーチアシスタントです。
- 結論を先に5行で要約
- 根拠となる文章・出典ファイル名を併記
- 「推測/仮定」は明確にラベリング
- 最後に次の調査質問5つを提案
```

**翻訳エージェント**（`.claude/agents/translator.md`）：

```yaml
---
name: translator
description: ファイル/文章を指定言語に翻訳し、マークダウン構造を維持します。
tools: Read, Write
model: sonnet
---

翻訳ルール：
- 固有名詞/製品名/コード/コマンドは原文のまま維持
- 文章のトーンは「丁寧だが大げさにしない」
- 表/目次/リンク構造は維持
```

**最終チェック（QA）エージェント**（`.claude/agents/editor-qa.md`）：

```yaml
---
name: editor-qa
description: 最終ドキュメントの誤字脱字・表記統一・禁止語・リンク漏れを点検します。
tools: Read, Grep, Glob
model: haiku
---

点検チェックリスト：
- 日付/数値表記の統一（例: 2026-02-26）
- 禁止語（完璧/革命/絶対）の検出
- リンク（https://）の漏れ・破損が疑われる箇所の指摘
- 「次のアクション」が抜けている場合は追加提案
```

<summary>🖥️ 開発者向けエージェント例（code-reviewer）</summary>

`.claude/agents/code-reviewer.md`：

```yaml
---
name: code-reviewer
description: コードをレビューします。コード変更後の品質、セキュリティ、パフォーマンスの検討に自動で使用。
tools: Read, Grep, Glob, Bash

---

あなたはシニアコードレビュアーです。

実行時：
1. `git diff`で最近の変更内容を確認
2. 修正されたファイルに集中

レビューチェックリスト：
- コードの可読性と命名規則
- エラー処理の適切さ
- セキュリティ脆弱性（露出したシークレット、入力バリデーション）
- テストカバレッジ

フィードバック形式：
- 🔴 Critical（必ず修正）
- 🟡 Warning（修正推奨）
- 🔵 Suggestion（検討事項）
```

## サブエージェントの設定オプション

```yaml
---
name: my-agent           # 小文字 + ハイフン（必須）
description: 説明        # Claudeがいつ委任するか判断する基準（必須）
tools: Read, Grep, Glob  # 許可ツール一覧（省略時はすべてのツールを継承）
disallowedTools: Write   # 除外するツール
model: haiku             # haiku / sonnet / opus / inherit（デフォルト）
permissionMode: default  # default / acceptEdits / dontAsk / bypassPermissions
maxTurns: 20             # 最大実行ターン数
memory: user             # 永続メモリ: user / project / local
background: false        # trueなら常にバックグラウンド実行
isolation: worktree      # worktreeなら隔離されたGitワークツリーで実行
---
```

## 実用例

### 例1: 並列リサーチ（競合の同時分析）

```
"競合A、B、Cをそれぞれ別のエージェントで同時に分析して。
 各エージェントはWebサイト、価格、レビューを調査して
 結果を一つの比較表に統合して"
```

→ 3つのリサーチエージェントが同時に作動
→ 結果を1つの比較表に統合
→ 順番に処理した場合と比べて3倍速

### 例2: ドキュメントの一括処理（PDF20個の並列要約）

```
"Downloadsフォルダの20個のPDFを5個ずつ分けて
 4つのエージェントが同時に要約し
 結果を1つのレポートにまとめて"
```

→ 4つの要約エージェントが並列実行、処理速度4倍
→ 膨大な要約ログがメインの会話を汚染しない

### 例3: 役割分担 — リサーチ → 編集の連鎖

```
"researcherエージェントでsources/フォルダを読んで1ページの要約を作って"
"editor-qaエージェントでfinal/フォルダのドキュメント品質を点検し、修正が必要な項目だけリストアップして"
"translatorエージェントでdrafts/post.mdを英語に翻訳してdrafts/post.en.mdとして保存して"
```

→ 各担当エージェントが順番に作業
→ メインのClaudeは流れの調整のみ

## 永続メモリ機能

サブエージェントに`memory`フィールドを設定すると、セッションをまたいでも学習内容が維持されます：

```yaml
---
name: code-reviewer
description: コードをレビューします
memory: user
---

コードをレビューしながら発見したコードベースのパターン、アーキテクチャ決定、繰り返し発生するイシューを
メモリに記録してください。次のレビュー時に以前の学習内容を活用してください。
```

| スコープ | 場所 | 使用場面 |
|--------|------|----------|
| `user` | `~/.claude/agent-memory/<名前>/` | 複数プロジェクトで知識を蓄積 |
| `project` | `.claude/agent-memory/<名前>/` | プロジェクト別の学習、チーム共有可能 |
| `local` | `.claude/agent-memory-local/<名前>/` | プロジェクト別の学習、個人のみ |

## フォアグラウンド vs バックグラウンド実行

- **フォアグラウンド**: 完了まで待機。権限リクエストがユーザーに伝達される
- **バックグラウンド**: バックグラウンドで同時実行。引き続き他の作業が可能

```
# バックグラウンド実行のリクエスト
"これをバックグラウンドで実行して"

# 実行中にバックグラウンドに切り替え
Ctrl+B
```

> **注意**: バックグラウンドのサブエージェントはMCPツールを使用できません。

## Agent Teamsとは？

**Agent Teams**はサブエージェントの一歩先を行く概念です。複数のエージェントがそれぞれ完全に独立したセッションで実行され、協力します。

| | Sub-agents | Agent Teams |
|---|---|---|
| **実行単位** | メインセッション内部 | 完全に独立したセッション |
| **コンテキスト** | メインと共有 | それぞれ独立 |
| **適した作業** | 単一セッション内での委任 | 長時間の並列作業 |
| **通信** | 結果要約を返す | セッション間の調整 |

> サブエージェントではコンテキストウィンドウが足りない場合や、作業が複数セッションにわたって持続する必要がある場合にAgent Teamsを検討します。

### プラグインで簡単に始める：キリキリ

Agent Teamsを直接設定するのが難しければ、**キリキリ（kkirikkiri）** プラグインが代わりにやってくれます。

```
/kkirikkiri リサーチチームを作って
```

自然言語の一言で：
1. **意図の把握** — リサーチ/開発/分析/コンテンツの中からマッチング
2. **インタビュー** — 2〜3個の質問で詳細な要件を収集
3. **環境スキャン** — インストール済みCLI（Codex、Geminiなど）を自動検出
4. **チーム構成の提案** — 役割別エージェント + 予想コスト/時間を案内
5. **実行** — Agent Teamsを自動生成・実行
6. **品質検証** — 結果を自動判定、不足があればリトライ

| プリセット | トリガー例 | 基本構成 |
|--------|-----------|----------|
| リサーチ | 「調べて」「探して」 | リーダー（Opus） + リサーチャー2名 |
| 開発 | 「作って」「実装して」 | リーダー（Opus） + 開発者2名 |
| 分析 | 「分析して」「把握して」 | リーダー（Opus） + 探索者2名 |
| コンテンツ | 「書いて」「ドキュメント作って」 | リーダー（Opus） + 作成者 + レビュアー |

> リーダーは常に最も賢いモデル（Opus）が担当し、コードは書かず計画・分配・検証のみ行います。

キリキリのインストール：プラグインインストールセクションのgptaku_pluginsを参照

## ⚠️ コストに注意

サブエージェントはそれぞれ独立したClaudeインスタンスです。**エージェントごとにトークンを消費します。**

- サブエージェント3つの並列実行 = トークン使用量が大幅に増加
- 結果がメインコンテキストに返る際に追加トークンを消費
- 不必要なサブエージェントの乱用はコスト増加につながる

**コスト削減のヒント**：
- 読み込み専用の探索には**Haikuモデル**のサブエージェントを使用
- サブエージェントが返す結果の量を明確に指定
- 単純な作業はメインの会話で処理

## 初心者へのアドバイス

> **最初は単一のClaudeで十分です。**

サブエージェントは以下の状況が発生した時に導入してください：
1. メインコンテキストが頻繁にあふれて圧縮が繰り返される時
2. 同じ探索作業を何度も繰り返す時
3. 特定のツールだけを使う安全なレビュアーが必要な時

---

## 一言まとめ

> Sub-agents = Claudeが別のClaudeに作業を委任。独立コンテキストで実行されるため、メインの会話をクリーンに保ちながら並列処理が可能。ただし、トークンコストはエージェント数に比例して増加。
